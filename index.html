WKWebView长按导致的问题

如果项目是window -> vc -> present tabController这样一种结构，在tabController中有一个webView，长按网页会出现UIAlertController,这时候做任何操作都会导致 tabController直接dismiss，问题的诡异在于这个dismiss的方法无法调试。

解决方案：
在info.plist 中添加相册读写的权限就可以避免这样的问题，可以想到的一点是如果网页中有图片，长按会出现保存图片的Alert，所以相册读写的权限也是有必要的。

ps:在与其他小伙伴交流的过程中发现如果是以window -> nav -> present tabController 这样的结构却不会出现这样的问题，

监听ScrollView停止滚动

在水群中看到某个小伙伴提出这样一个需求，在不用UIScrollViewDelegate的情况下如何知道ScrollView停止了滚动呢，很多人看到这里脑子里想到的第一个方案肯定是KVO,监听 contentOffset的值变化，然后通过新旧值的变化来判断是否停止滚动哈。

这里我们先来谈谈ScrollView 停止滚动有多少种情况：
1、划一下，自然停止滚动；
2、当ScrollView正在快速滚动时，手指按住屏幕停止滚动；
3、手指不离开屏幕，慢速上下滑动停止。

上面三种情况，如果单纯用contentOffset的值变化来判断恐怕是不能够完成(信不信由你，反正我没用试过的)，毕竟当中还涉及到scrollView的bounces、deceleration等特性。那么怎么才能很好的抓住视图刚刚停止滚动的那个点呢。大家不妨想想NSTimer 和 UIScrollView，造成倒计时不跑的原因不就是因为scrollView在滚动嘛，视图不滚动了倒计时就开始跑啦，这不就说明已经有一个很好的切入点来知道ScrollView是否在滚动了吗。

找到问题的关键点就好办了，我们都知道NSTimer不被调用是因为在视图滚动时Runloop切换成了UITrackingRunloopMode，所以我们只要在UITrackingRunloopMode 退出时就可以知道scrollView 恰好停止了滚动。

两个概念 shell && shell脚本

shell

shell 是一种应用程序，这个程序提供一个洁面，用户通过这个界面访问操作系统内核的服务，sh 是一种Unix shell， Windows Explore 是一个典型的图形界面

shell 脚本

shell脚本(shell script) 是一种为shell 编写的脚本程序，通常所说的shell 通常是指shell 脚本

环境

shell 编程和其他语言一样，只要有一个写代码的文本编辑器，还有一个解释执行脚本的解释器就可以了

Mac OS不仅带有sh、bash 这两个最基本的解释器，还有ksh、csh、zsh等不常用的解释器

sh

即Bourne shell，POSIX（Portable Operating System Interface）标准的shell 解释器，它的二进制文件路径通常是/bin/sh，这个解释器由Bell Labs （贝尔实验室）开发

bash

Bash 是Bourne shell 的替代品，属于GNU Project，二进制文件路径通常是/bin/bash

编程语言

理论上说，只要一门语言提供了解释器（而不仅是编译器），这门语言就可以胜任脚本编程，常见的解释型语言都可以用来脚本编程，如Perl、Tcl、Python、PHP、Ruby
编译型语言，只要有解释器，也可以用作脚本编程，如C shell是内置的（／bin/csh）

关于Runloop的一点笔记

基本介绍

Runloop本质是一个‘do while’循环，解决的问题就是让程序在接受到任务时就开始执行任务，在没有任务时处于休眠状态下；从App层面上来看，一般app处于前台那么就会处于执行任务的状态；app处于后台那么就会处于休眠状态；从代码层面上来看，我们都知道一个App可以看成一个进程，而进程的最小单位是线程，每个线程在执行完后是会退出的。那么Runloop这样的运行模式与线程之间的关系就很好理解了。
